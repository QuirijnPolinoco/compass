# Scaling: indexing a large repo

> Question: does Compass handle a **100k-file** repo, and where does the time go?
> Honest by design — every number below is measured on the machine noted, nothing is invented.

## TL;DR

- **Parsing/extraction is not the bottleneck.** Parse + extract for **101,000 files** is **~1s of
  CPU** (16 cores) — sub-millisecond per file. The graph assemble, call resolution, and import
  resolution are together **under 0.4s**.
- **The cost of a first index is first-read file I/O** — opening and reading 101k files cold
  (amplified on Windows, where Defender scans every open). That was **~74s** here.
- **Incremental indexing removes it on every later index.** A per-file extraction cache
  (`.compass/extractions.json`, keyed by `mtime`/`size`) lets unchanged files skip the read+parse
  entirely. After the first index, re-indexing **101k files drops parse+extract from ~970ms to
  ~20ms**, and editing one file re-reads only that file.

## Setup

- **Repo:** a synthetic Python tree — 1,000 packages × 100 modules = **101,000 files** (incl.
  `__init__.py`), each module importing a sibling and a module in the next package (~200k import
  edges). Generated by the script in *Reproducing* below.
- **Build:** `cargo build --release` (the shipped artifact: LTO, 1 codegen unit).
- **Machine:** Windows 11, 16 logical cores, NVMe SSD, Windows Defender real-time protection on.
- **Method:** `COMPASS_TIMING=1 compass init <repo>` prints per-phase wall-clock to stderr.

## Results

Per-phase wall-clock (`COMPASS_TIMING=1`):

| Phase | First index (cold I/O) | First index (warm OS cache) | Re-index, no changes |
|-------|------:|------:|------:|
| walk | 0.26s | 0.26s | 0.25s |
| **parse+extract** | **73.8s** | **0.97s** | **0.02s** |
| assemble | 0.11s | 0.07s | 0.06s |
| resolve-calls | 0.01s | 0.01s | 0.01s |
| build-indices | 0.05s | 0.04s | 0.03s |
| resolve-imports | 0.27s | 0.21s | 0.23s |
| **total `init`** | **~75s** | **~2.7s** | **~1.6s** |

Reading the table:

- **parse+extract 73.8s → 0.97s** between the cold and warm runs (same code, same parallelism) is
  the proof that the 73.8s is **file-read I/O**, not CPU. Single-threaded warm parse+extract was
  8.0s, so rayon already gives ~8× — parsing is well-parallelized.
- **parse+extract 0.97s → 0.02s** between the first and second index is **incremental reuse**: the
  second run reused all 101k cached extractions instead of re-reading them. The resulting graph is
  identical (100,000 symbols, 200,000 import edges).
- **Editing one file** then re-indexing: parse+extract **~0.02s**, and the new symbol appears
  (100,000 → 100,001) — only the edited file was re-read.

## What this means

- Compass **scales to 100k files**: the per-file work is cheap and parallel; the only large cost is
  reading the tree the first time, which is inherent (you must read every file once) and dominated
  by the OS/AV on a cold cache.
- After the first index, **`watch` and repeated `init`/`map` are near-instant** because unchanged
  files are served from the extraction cache. This is the realistic developer loop on a big repo.

### Caveats (so you can trust the rest)

- **n = 1**, one machine, a **synthetic** repo (small files, regular imports). Real repos have
  larger files and messier graphs; absolute numbers will differ. The *shape* (parse is cheap; I/O
  and incremental reuse dominate) is the durable finding.
- The cold 73.8s is **Windows + Defender-heavy**. On Linux/macOS without per-open AV scanning, a
  cold first index of the same tree is substantially faster. The incremental win applies on all
  platforms.
- Change detection uses **`mtime` + `size`**, not a content hash (hashing would require the read we
  are trying to avoid). A pathological same-size, same-mtime edit could be missed — delete
  `.compass/` (or it auto-invalidates on a cache-format bump) to force a full re-index.

## Reproducing

Generate the repo (Python), then index it twice:

```python
# gen.py — writes N modules under <root>/src
import os, sys
root, N, PER = sys.argv[1], int(sys.argv[2]), 100
base = os.path.join(root, "src"); os.makedirs(base, exist_ok=True)
pkgs = (N + PER - 1) // PER; count = 0
for p in range(pkgs):
    d = os.path.join(base, f"pkg{p}"); os.makedirs(d, exist_ok=True)
    open(os.path.join(d, "__init__.py"), "w").close()
    for m in range(PER):
        if count >= N: break
        sib, nxt = (m + 1) % PER, (p + 1) % pkgs
        with open(os.path.join(d, f"mod{m}.py"), "w") as fh:
            fh.write(f"from pkg{p}.mod{sib} import f{sib}\nfrom pkg{nxt}.mod0 import f0\n\ndef f{m}():\n    return {m}\n")
        count += 1
    if count >= N: break
```

```sh
python gen.py /tmp/bigrepo 100000
COMPASS_TIMING=1 compass init /tmp/bigrepo   # first index — builds .compass/extractions.json
COMPASS_TIMING=1 compass init /tmp/bigrepo   # re-index — reuses the cache (parse+extract collapses)
```
